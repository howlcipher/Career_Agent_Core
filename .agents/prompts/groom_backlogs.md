# Groom the Backlogs

Maintenance pass over `bugs.md`, `improvements.md`, and `documentation/task_journals/`. Do not implement any backlog item during this pass.

**Subagents and delegation:** when re-verification fans out across many items or files, you may parallelize it with subagents or delegate it, subject to the delegation policy in `work_next_item.md`: read-only verification subagents (e.g. Claude Code's Explore agent) are allowed but bill the same plan as the session, so for limit-saving prefer headless Antigravity (`agy -p`) or local Ollama for self-contained verification briefs. All edits to the backlog documents, rankings, and journals stay in this session, and every claim a subagent or delegate reports must be spot-checked (grep the actual code) before it changes a row.

**Model fallback:** the models named in the backlog tables are suggestions, not requirements. Probe availability live before delegating and walk down this ladder until something works: (1) the item's recommended model for the host CLI you are running in; (2) other tiers on the same subscription (Antigravity: `agy models`, quota is shared across Gemini tiers; Claude Code: step down Sonnet → Haiku); (3) local Ollama tags (`curl localhost:11434/api/tags`); (4) the session's own model doing the work inline. Only report the pass blocked if every rung fails — a groom is mostly reading and editing markdown, so any available model is acceptable.

1. **Check the Usability Gate** at the top of `bugs.md` first. Verify each checked box is still true (re-run `go build ./...`, `go vet ./...`, `go test ./...` for the three static ones) and check whether any of the four live end-to-end boxes can now be verified or has new information against it. Update the Status line accordingly.
2. Re-evaluate every Pending row in both documents against the *current code*, not the row's prose — grep for the relevant package/function before trusting a description. This backlog has drifted from the code before (three "planned" items turned out to already be shipped). Update, merge, or close stale items with a dated note; keep table rows and detail sections in sync.
3. Re-score and re-rank every Pending row using the diminishing-returns model defined in `improvements.md`: **Score = (Value 1–8 × Decay) ÷ Effort 1–8**, where Decay halves for each already-shipped item in the same theme and stays 1.0 for new-capability items. Recompute all scores here rather than trusting last groom's numbers.
4. Apply the **ROI floor (0.5)**: flag every Pending row scoring below it with `⚠️ below floor` and a dated scoring note. Below-floor items stay open — never close one unilaterally — but must not be worked without explicit user confirmation; surface each one to the user with a recommendation (confirm, re-scope, or close). Clear the flag when a re-score lifts an item back above the floor.
5. Journals: delete any journal whose item is no longer outstanding, and consolidate any journal too large to skim, preserving its Next Step line.
6. Commit the grooming as `docs(backlog): <summary>` and push.
